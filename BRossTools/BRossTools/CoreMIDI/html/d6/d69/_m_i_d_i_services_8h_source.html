17 /*!
18  @header MIDIServices.h
19 
20  This is the header file for Mac OS X's MIDI system services.
21 
22  API Overview
23  ------------
24 
25  Drivers own and control Devices, e.g. USB interfaces, PCI cards, etc. A device is defined
26  as a physical object that would be represented by a single icon if there were a graphical
27  view of the studio.
28 
29  Devices may have multiple logically distinct sub-components, e.g. a MIDI synthesizer and a
30  pair of MIDI ports, both addressable via a USB port. These are called Entities.
31 
32  Entities have any number of Endpoints, sources and destinations of 16-channel MIDI streams.
33  By grouping a device's endpoints into entities, the system has enough information for an
34  application to make reasonable default assumptions about how to communicate in a
35  bi-directional manner with each entity, as is necessary in MIDI librarian applications.
36 
37  CoreMIDI attaches a collection of properties to each object it manages. Some properties are
38  dynamic characteristics of a device (e.g. MIDI receive channel and system-exclusive ID's),
39  and some are a matter of user preference (choice of icon, whether the device should appear
40  in lists of possible controllers). Other properties are static and could be looked up in a
41  database, using the device's manufacturer and model names as a key.
42 
43  Implementation overview
44  -----------------------
45 
46  The client API is implemented as the CoreMIDI framework, which uses IPC to communicate with
47  a server process, MIDIServer.
48 
49  The server process loads, and manages all communication with, MIDI drivers.
50 
51  "Drivers" are not I/O Kit drivers. They are bundles containing CFPlugins.
52 
53  Many MIDI drivers can simply be user-side I/O Kit clients (probably for serial, USB,
54  Firewire).
55 
56  PCI card drivers will need their MIDI drivers to communicate with a separate kernel
57  extension.
58 
59  Note about CoreFoundation data types (CFString, CFData, CFDictionary)
60  ---------------------------------------------------------------------
61 
62  When passing a CF object to a MIDI function, the MIDI function will never consume a
63  reference to the object; the caller always retains a reference which it is responsible for
64  releasing with CFRelease().
65 
66  When receiving a CF object as a return value from a MIDI function, the caller always
67  receives a new reference to the object, and is responsible for releasing it.
68 */
