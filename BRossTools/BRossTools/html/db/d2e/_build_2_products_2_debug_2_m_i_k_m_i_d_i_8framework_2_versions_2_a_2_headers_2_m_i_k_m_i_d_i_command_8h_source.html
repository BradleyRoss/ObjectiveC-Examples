62 /**
63  * In MIKMIDI, MIDI messages are objects. Specifically, they are instances of MIKMIDICommand or one of its
64  * subclasses. MIKMIDICommand's subclasses each represent a specific type of MIDI message, for example,
65  * control change command messages will be instances of MIKMIDIControlChangeCommand.
66  * MIKMIDICommand includes properties for getting information and data common to all MIDI messages.
67  * Its subclasses implement additional method and properties specific to messages of their associated type.
68  *
69  * MIKMIDICommand is also available in mutable variants, most useful for creating commands to be sent out
70  * by your application.
71  *
72  * To create a new command, typically, you should use +commandForCommandType:.
73  *
74  * Subclass MIKMIDICommand
75  * -----------------------
76  *
77  * Support for the various MIDI message types is provided by type-specific subclasses of MIKMIDICommand.
78  * For example, Control Change messages are represented using MIKMIDIControlChangeCommand. MIKMIDI
79  * includes a limited number of MIKMIDICommand subclasses to support the most common MIDI message types.
80  * To support a new command type, you should create a new subclass of MIKMIDICommand (and please consider
81  * contributing it to the main MIKMIDI repository!). If you implement this subclass according to the rules
82  * explained below, it will automatically be used to represent incoming MIDI commands matching its MIDI command type.
83  *
84  * To successfully subclass MIKMIDICommand, you *must* override at least the following methods:
85  *
86  * - `+supportedMIDICommandTypes:` - Return an array of one or more MIKMIDICommandTypes that your subclass supports.
87  * - `+immutableCounterPartClass` - Return the subclass itself (eg. `return [MIKMIDINewTypeCommand class];`)
88  * - `+mutableCounterPartClass` - Return the mutable counterpart class (eg. `return [MIKMIDIMutableNewTypeCommand class;]`)
89  *
90  * Optionally, override `-additionalCommandDescription` to provide an additional, type-specific description string.
91  *
92  * You must also implement `+load` and call `[MIKMIDICommand registerSubclass:self]` to register your subclass with
93  * the MIKMIDICommand machinery.
94  *
95  * When creating a subclass of MIKMIDICommand, you should also create a mutable variant which is itself
96  * a subclass of your type-specific MIKMIDICommand subclass. The mutable subclass should override `+isMutable`
97  * and return YES.
98  *
99  * If your subclass adds additional properties, beyond those supported by MIKMIDICommand itself, those properties
100  * should only be settable on instances of the mutable variant class. The preferred way to accomplish this is to
101  * implement the setters on the *immutable*, base subclass. In their implementations, check to see if self is
102  * mutable, and if not, raise an exception. Use the following line of code:
103  *
104  * if (![[self class] isMutable]) return MIKMIDI_RAISE_MUTATION_ATTEMPT_EXCEPTION;
105  *
106  * For a straightforward example of a MIKMIDICommand subclass, see MIKMIDINoteOnCommand.
107  *
108  */
