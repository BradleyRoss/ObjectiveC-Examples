24 /*!
25  @header MIDIDriver.h
26 
27  This is the header file for Mac OS X's MIDI driver interface.
28 
29 
30  MIDI drivers are CFPlugIns, installed into the following places:
31 
32  ```
33  /System/Library/Extensions -- not recommended for non-Apple drivers, but
34  necessary for compatibility with CoreMIDI 1.0
35 
36  /Library/Audio/MIDI Drivers -- starting with CoreMIDI 1.1
37 
38  ~/Library/Audio/MIDI Drivers -- starting with CoreMIDI 1.1
39  ```
40 
41  Refer to the CFPlugIn documentation for more information about plug-ins.
42 
43 
44  A driver's bundle settings should include settings resembling the following:
45 
46  ```
47  Bundle settings:
48  CFBundleIdentifier String com.mycompany.midi.driver.mydevice
49  (note that this will be the driver's persistent ID in MIDISetup's)
50  CFPlugInDynamicRegistration String NO
51  CFPlugInFactories Dictionary 1 key/value pair
52  [your new factory UUID] String [your factory function name]
53  CFPlugInTypes Dictionary 1 key/value pair
54  ECDE9574-0FE4-11D4-BB1A-0050E4CEA526 Array 1 object
55  (this is kMIDIDriverTypeID)
56  0 String [your new factory UUID]
57  Build settings:
58  WRAPPER_EXTENSION plugin
59  ```
60 
61 
62  Drivers have access to most of the CoreMIDI API. Starting in Mac OS X 10.6, drivers should link
63  with CoreMIDI.framework. In previous versions of Mac OS X, drivers should link with
64  CoreMIDIServer.framework, not CoreMIDI.framework.
65 
66  On Mac OS X versions prior to 10.6, MIDI driver plugins linked against the CoreMIDIServer
67  framework in order to access the CoreMIDI API. Drivers which are to run on earlier OS versions
68  should be built 32-bit fat (ppc and i386) and link against CoreMIDIServer. Drivers which are to
69  run on Mac OS X 10.6 and later should be built for x86_64 and link against the CoreMIDI
70  framework. Drivers which are to run on all versions of Mac OS X should be build 3-way fat (ppc,
71  i386, and x86_64), with the ppc and i386 slices linking against CoreMIDIServer, and the x86_64
72  slice linking against CoreMIDI.
73 
74  Unlike applications, drivers communicate with the server directly, not through Mach messaging.
75  This necessitates some limitations on the contexts from which a driver may call the server.
76 
77  The MIDI I/O functions MIDISend and MIDIReceived may be called from any thread.
78 
79  All other CoreMIDI functions must only be called from the server's main thread, which is the
80  thread on which the driver is created and from which all calls to the driver other than Send()
81  are made.
82 */

125  detect the devices which are present. For each device present, the driver should
126  create a MIDIDeviceRef with entities, using MIDIDeviceCreate and
127  MIDIDeviceAddEntity, and add the device to the supplied MIDIDeviceListRef, using

134  /*!
135  @fn Start
136  @discussion
137  The server is telling the driver to begin MIDI I/O.
138 
139  The provided device list contains the devices which were previously located by
140  FindDevices (in the case of a version 1 driver), or the devices which are owned by
141  this driver and are currently in the current MIDISetup (for version 2 drivers).
142 
143  The provided devices may or may not still be present. A version 1 driver should
144  attempt to use as many of the devices as are actually present.
145 
146  A version 2 driver may make calls such as MIDISetupAddDevice, MIDIDeviceAddEntity,
147  MIDIDeviceRemoveEntity to dynamically modify the system's current state. For devices
148  in the provided device list which are not present, the driver should set their
149  kMIDIPropertyOffline property to 1. A version 2 driver may also set up
150  notifications when the IORegistry changes, to detect connection and disconnection of
151  devices it wishes to control. At these times also, the driver may change the
152  devices' kMIDIPropertyOffline, and dynamically modify the system's current state to
153  reflect the devices which are present. When passing a CFRunLoopRef to IOKit for
154  notification purposes, the driver must use the server's main runloop, which is
155  obtained with CFRunLoopGetCurrent().
156 
157  The driver will probably want to iterate through the destination endpoints and
158  assign their driver refCons, so as to identify multiple destinations when Send() is
159  called.
160 
161  The provided device list remains owned by the system and can be assumed to contain
162  only devices owned by this driver. The driver may retain references to the devices
163  in this list and any it creates while running.
164  */
