18 /*!
19  @header MIDICapabilityInquiry.h
20 
21  This is the header file for MIDI Capability Inquiry (MIDI-CI) system services.
22 
23  API Overview
24  ------------
25 
26  MIDI-CI is the portion of MIDI 2.0 that allows for the selection of data protocol and the
27  advertisement and manipulation of profiles and properties. These three capabilities are
28  sometimes referred to as the three P's. MIDI 2.0 (and specifically MIDI-CI) are bidirectional
29  protocols, unlike MIDI 1.0 which could be unidirectional or bidirectional.
30 
31  The two sides of MIDI-CI communication are the Initiator and Responder. Most MIDI-CI messages
32  can only be sent from the Initiator or from the Responder, but a small number of messages could
33  originate from the Initiator or the Responder.
34 
35  MIDI client processes playing the Initiator roles of a MIDI-CI connection use the MIDICISession
36  API, which allows for profile and property manipulation on the associated responder. MIDI client
37  processes that wish to advertise their own profile and property capabilities use the MIDICIResponder
38  API, which is capable of sending Responder-originated MIDI-CI messages to Initiators.
39 
40  Implementation overview
41  -----------------------
42 
43  With MIDI-CI beta enabled, Universal MIDI System Exclusive (SysEx) messages are filtered from the
44  I/O stream and sent to the MIDI-CI subsystem for routing and handling. Other SysEx messages are
45  untouched. In the release API, filtering will be on by default, and Universal MIDI SysEx messages
46  with associated CoreMIDI API will only be received and routed by the MIDI-CI subsystem.
47 
48  Discovering MIDI-CI capable nodes managed by CoreMIDI is accomplished with MIDICIDiscoveryManager,
49  which sends a MIDI-CI discovery message to all MIDI destinations in the system and aggregates the
50  respondents as MIDICIDiscoveredNode objects.
51 
52  After MIDI-CI discovery has been performed, a MIDI client process may act as Initiator by creating
53  a MIDICISession using a MIDICIDiscoveredNode. The MIDICISession is synchronously returned prior to
54  capability inquiry, and the supplied handler is invoked once the results are received or after
55  timeout. Any available profiles may then be examined/toggled for each channel and the whole port
56  (represented by MIDI channel 0x7F).
57 
58  No API for protocol negotiation is currently supplied. CoreMIDI is MIDI 2.0-capable system-wide
59  and will not only select the appropriate protocol for connected hardware but will also automatically
60  convert between MIDI 1.0 and MIDI 2.0 protocol as needed. See MIDIServices.h for API guidelines and
61  usage for MIDI 2.0 protocol and Universal MIDI Packets (UMP).
62 
63  Additionally, no explicit property exchange API is provided. However, property exchange can be
64  accomplished by performing first performing MIDI-CI discovery and then sending the appropriate
65  Universal MIDI SysEx messages to the supplied destination and monitoring the MIDICISession's
66  supplied source. Responses to Has Property, Get Property, Set Property, and Subscribe to
67  Property messages are not filtered by the MIDI-CI subsystem and can be received by any MIDI client.
68  However, Property Exchange capabilities must still be discovered using API first since the
69  Inquiry: Property Exchange Capabilities (msg ID 0x30) and Reply to Property Exchange Capabilities
70  (msg ID 0x31) are reserved by CoreMIDI.
71 
72  A MIDI client process may also/instead play the role of responder by creating a MIDICIResponderDelegate
73  and then a MIDICIResponder. Similarly to MIDICISession, a MIDICIResponder may advertise property
74  capabilities, but the details of property exchange rely on the implementation.
75 
76  Please visit http://www.midi.org/specifications for more information on MIDI Capability Inquiry
77  and a list of currently defined capabilities.
78 */
