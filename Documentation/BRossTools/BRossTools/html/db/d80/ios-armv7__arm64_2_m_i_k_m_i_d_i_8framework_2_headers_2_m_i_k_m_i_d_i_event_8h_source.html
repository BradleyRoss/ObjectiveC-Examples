75 /**
76  * In MIKMIDI, MIDI events are objects. Specifically, they are instances of MIKMIDIEvent or one of its
77  * subclasses. MIKMIDIEvent's subclasses each represent a specific type of MIDI event, for example,
78  * note events will be instances of MIKMIDINoteEvent.
79  * MIKMIDIEvent includes properties for getting information and data common to all MIDI events.
80  * Its subclasses implement additional method and properties specific to messages of their associated type.
81  *
82  * MIKMIDIEvent is also available in mutable variants, most useful for creating events to be sent out
83  * by your application.
84  *
85  * To create a new event, typically, you should use +midiEventWithTimeStamp:eventType:data:(NSData *)data
86  *
87  * Subclass MIKMIDIEvent
88  * -----------------------
89  *
90  * Support for the various MIDI event types is provided by type-specific subclasses of MIKMIDIEvent.
91  * For example, note events are represented using MIKMIDINoteEvent.
92  *
93  * To support a new event type, you should create a new subclass of MIKMIDIEvent (and please consider
94  * contributing it to the main MIKMIDI repository!). If you implement this subclass according to the rules
95  * explained below, it will automatically be used to represent MIDI events matching its MIDI event type.
96  *
97  * To successfully subclass MIKMIDIEvent, you *must* override at least the following methods:
98  *
99  * - `+supportsMIKMIDIEventType:` - Return YES when passed the MIKMIDIEventType value your subclass supports.
100  * - `+immutableCounterPartClass` - Return the subclass itself (eg. `return [MIKMIDINewTypeEvent class];`)
101  * - `+mutableCounterPartClass` - Return the mutable counterpart class (eg. `return [MIKMIDIMutableNewTypeEvent class;]`)
102  *
103  * Optionally, override `-additionalEventDescription` to provide an additional, type-specific description string.
104  *
105  * You must also implement `+load` and call `[MIKMIDIEvent registerSubclass:self]` to register your subclass with
106  * the MIKMIDIEvent machinery.
107  *
108  * When creating a subclass of MIKMIDIEvent, you should also create a mutable variant which is itself
109  * a subclass of your type-specific MIKMIDIEvent subclass. The mutable subclass should override `+isMutable`
110  * and return YES.
111  *
112  * If your subclass adds additional properties, beyond those supported by MIKMIDIEvent itself, those properties
113  * should only be settable on instances of the mutable variant class. The preferred way to accomplish this is to
114  * implement the setters on the *immutable*, base subclass. In their implementations, check to see if self is
115  * mutable, and if not, raise an exception. Use the following line of code:
116  *
117  * if (![[self class] isMutable]) return MIKMIDI_RAISE_MUTATION_ATTEMPT_EXCEPTION;
118  *
119  * For a straightforward example of a MIKMIDIEvent subclass, see MIKMIDINoteEvent.
120  *
121  */
